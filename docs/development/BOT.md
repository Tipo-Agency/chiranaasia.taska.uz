# Telegram-бот: цели, архитектура, конфигурация и план внедрения

Документ описывает **как бот должен работать** (личка vs группа), **как это согласуется с backend/Redis/БД**, и **что уже есть в репозитории**. Реализация в `apps/bot` постепенно приводится к этой модели.

---

## 1. Личка и группа: противоречит ли это логике?

**Нет.** Один и тот же бот (один токен у BotFather) может одновременно:

- получать обновления из **личных чатов** (пользователь ↔ бот);
- получать обновления из **групп и супергрупп**, куда бот добавлен.

Telegram различает контексты по **`chat_id`** (отрицательный ID — обычно группа/канал; положительный — личка). Поведение в коде задаётся правилами: *куда отвечать*, *какие команды доступны в группе*, *куда слать системные уведомления*.

Итого:

| Режим | Назначение (целевое) |
|--------|----------------------|
| **Личка** | Полноценное взаимодействие с системой под авторизованным пользователем: задачи, сделки, процессы, финансы, документы, недельные планы, просмотр и действия. |
| **Группа** | Общие уведомления по событиям (по настройкам), дайджесты, опционально команды для командной работы (создать задачу из reply, запросить список сущностей). |

Это стандартная схема для корпоративных ботов: **личка = рабочее место**, **группа = витрина/коллективный канал**.

## 2. Как организовать конфигурацию (токен, группа, пользователи)

### 2.1 Токен бота

- **Где хранится в проде:** секрет окружения (например `TELEGRAM_BOT_TOKEN` / `TELEGRAM_EMPLOYEE_BOT_TOKEN` в API и в сервисе `apps/bot`), не в репозитории.
- **Веб-настройки «токен бота»** (если вводят в UI): имеет смысл только как **удобная подстановка в демо** или с сохранением на сервере в защищённом хранилище; для production токен задаётся **на стороне сервера**, чтобы не светить его в браузере и в лишних логах.

Один токен = один активный long polling / webhook (см. раздел про 409 в конце).

### 2.2 Личные сообщения: привязка пользователя

Чтобы бот понимал **какой пользователь CRM** пишет в личку, нужна связь:

- в профиле пользователя в системе хранится **`telegram_user_id`** (числовой ID из Telegram);
- при первом `/start` или коде связывания бот сохраняет эту связь через API.

Без этого бот не может безопасно показывать «мои задачи».

### 2.3 Группа: куда слать «общие» уведомления

В модели уведомлений уже заложено поле **`telegramGroupChatId`** (в `notification_preferences` / API `notification-prefs`): это **ID группы** (строка вида `-100…`), куда бот может отправлять сообщения, если:

- бот **добавлен в группу** и имеет право писать;
- в настройках для типов событий включён канал **«групповой Telegram»** (`telegramGroup` и т.д. — см. схему prefs в API).

Практика:

1. Создать группу, добавить бота, дать право на сообщения.
2. Узнать `chat_id` группы (через `@userinfobot`, лог бота или временную команду `getUpdates`).
3. Сохранить `telegramGroupChatId` в настройках уведомлений (организации/дефолтного пользователя prefs — в зависимости от того, как у вас задана модель «чей» prefs).

Таким образом: **да, в настройках логично иметь токен (на сервере) + ID группы (в prefs)** — это разные вещи: токен идентифицирует *бота*, `chat_id` — *куда* слать в группу.

## 3. Redis, события и БД: как лучше устроить поток

**Коротко:** Redis **не** должен быть местом, где бот «читает всю бизнес-логику». Источник правды по сущностям **PostgreSQL**. Redis **Streams** — транспорт шины событий; история событий и очередь доставок — в **БД**.

### 3.1 Как устроено в проекте (факт)

1. Доменное событие при создании/обновлении вызывает `emit_domain_event` (см. `apps/api/app/services/domain_events.py`):
   - в таблицу **`notification_events`** пишется запись;
   - событие публикуется в **Redis Stream** (`publish_domain_event`);
   - вызывается **`process_domain_event`** (notification hub), который создаёт записи уведомлений и доставок (`notifications`, `notification_deliveries`, зеркалирование в чат и т.д.).

2. Если Redis недоступен, событие **всё равно остаётся в PostgreSQL** (`published_to_stream` может быть false) — см. `NOTIFICATIONS_ARCHITECTURE.md`.

3. Доставка в Telegram по API реализована как **обработка pending-записей `notification_deliveries`** с каналом `telegram` (`run_pending_deliveries` в `apps/api/app/services/notification_delivery.py`) — т.е. **боту или отдельному воркеру не обязательно ходить в Redis**: достаточно периодически вызывать API/worker, который забирает pending и шлёт через Bot API.

### 3.2 Рекомендуемая целевая схема для Telegram

| Компонент | Роль |
|-----------|------|
| **PostgreSQL** | Сущности (задачи, сделки…), `notification_events`, `notifications`, `notification_deliveries`, prefs пользователя. |
| **Redis Streams** | Транспорт/масштабирование, аналитика, не для ручного чтения ботом «всего подряд». |
| **API** | Авторизация пользователя, CRUD, выдача «что отправить в Telegram», выполнение команд от бота. |
| **`apps/bot`** | Long polling/webhook, парсинг команд, сценарии лички, групповые команды; вызовы **HTTP API** backend. |
| **Доставка уведомлений в Telegram** | Либо внутри API cron/worker (`run_pending_deliveries`), либо тот же процесс, что и бот, **если** он читает только API/очередь из БД — но не Redis напрямую как основной интерфейс. |

Итог: **бот не обязан «читать Redis»**; правильнее — **читать события/доставки через API** или полагаться на **worker**, который отправляет сообщения ботом. Redis остаётся для шины и realtime в других частях системы.

## 4. Все ли события попадают в Redis?

**Почти все мутации через основные бизнес-роутеры — да.** Для типичных `PUT`/`POST`/`PATCH`/`DELETE` вызывается `log_entity_mutation` → `emit_domain_event`: запись в **`notification_events`**, попытка публикации в **Redis Stream**, затем `process_domain_event` (in-app уведомления и доставки создаются только для типов, известных notification hub).

Подключены (через `log_entity_mutation` / прямой `emit` где уже было): `tasks`, `deals`, `meetings`, `docs`, `clients`, `projects`, `employees`, `departments`, `tables`, `folders`, `content_posts`, `statuses`, `priorities`, `automation`, `bpm`, `inventory`, `accounts_receivable`, `funnels`, **`finance`** (категории, фонды, план, заявки, финплан/планирования, выписки, отчёты по приходам, БДР), **`weekly_plans`** (планы и протоколы), **`messages`** (отправка и прочтение), **`auth`** (пользователи), **`notification_prefs`**, **`activity`** (только `POST` добавления записи — без `PUT` синка, чтобы не плодить тысячи событий), **`notifications`** (смена прочитанности), плюс ручная публикация `POST /notification-events/publish`.

**Исключения и оговорки:**

- **Админка** (`/admin/*`): обслуживающие действия (requeue доставок, retention, прогон тестов) **не** логируются как доменные события.
- **Системные логи** (`/system/logs`) — только чтение.
- **`activity` `PUT`**: одно доменное событие **`activity_log.bulk_synced`** на весь запрос (количество строк в payload), без отдельного emit на каждую строку — иначе был бы шум при синхронизации тысяч записей.
- Любой **новый** роутер или фоновая задача без вызова `log_entity_mutation` по-прежнему **не** попадёт в шину, пока не подключат.
- При недоступном Redis событие остаётся в БД (`published_to_stream` может быть false) — как в `NOTIFICATIONS_ARCHITECTURE.md`.

Вывод: для **основного CRUD по продуктовым модулям** ожидание «событие в БД и при живом Redis — в стрим» справедливо; **100 % любых изменений в системе** (включая админские и legacy-пути) по-прежнему не гарантируется без явного вызова.

## 5. Функциональные требования (целевое видение)

### 5.1 Группа

- **Уведомления** по событиям (как в настройках: тип события, канал «группа», текст/шаблон).
- Опционально: **команды** (`/task`, `/deal`, `/help`) и **reply** к сообщению → создание задачи/сущности в системе от имени пользователя (с проверкой прав и `telegram_user_id`).
- Опционально: **короткие команды** для выборки «мои открытые задачи» / «сделки в воронке» с ограничением, чтобы не спамить (только для ролей или только для админов группы).

### 5.2 Личка

Полноценное меню и сценарии (без полного дублирования веб-UI, но с ключевыми действиями):

- задачи (список, фильтры, смена статуса, создание);
- сделки CRM;
- BPM (запуск процессов, шаги, где применимо);
- финансы (заявка на выделение средств и т.д. — по вашему ТЗ);
- документы;
- недельные планы;
- уведомления и напоминания.

Приоритизация — по этапам (MVP → расширение).

## 6. Связь с текущим кодом `apps/bot`

- **Архитектура (v3):** пакет `taska_bot/` — **асинхронный httpx** к `/api`, **`telegram.ext.JobQueue`** (тот же event loop, без отдельного APScheduler). Запуск: `python main.py` или `python bot.py` (тонкий shim для старых systemd-скриптов).
- **Команды:** `/start` (логин/пароль CRM), привязка `telegramUserId`, `/tasks` (в т.ч. по сохранённому ID после рестарта). Джобы: личные напоминания 9:00, сводка в группу 9:00, еженедельный отчёт по понедельникам 9:00, заявки/поздравления по сделкам каждые 5 мин (если задан `telegramGroupChatId` в prefs).
- Доставка системных уведомлений из CRM в Telegram по-прежнему через **`notification_deliveries`** в API; бот не читает Redis.

## 7. Деплой и эксплуатация

- Скрипт **`ops/scripts/deploy.sh`**, `telegram-bot.service`, `apps/bot/deploy.sh` — см. исторический раздел ниже.
- **Конфликт 409:** один токен = один активный процесс long polling/webhook. Не запускать два инстанса с одним токеном.

### Переменные окружения (`taska_bot/settings.py`, `.env`)

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | Обязательный токен бота. |
| `BACKEND_URL` | Базовый URL API без `/api` (например `http://127.0.0.1:8003`). |
| `DEFAULT_TIMEZONE` | По умолчанию `Asia/Tashkent` (джобы JobQueue). |

### Локальная разработка

В `docker-compose.yml` сервис бота по умолчанию может быть отключён — включайте осознанно.

## 8. План реализации (рекомендуемые этапы)

1. **Инвентаризация API:** какие операции уже есть для задач/сделок/пользователей; добавить недостающие тонкие эндпоинты для бота (создание задачи, привязка Telegram ID).
2. **Единая доставка уведомлений:** закрепить worker/cron `run_pending_deliveries` + токен бота в API; убрать дублирование с legacy очередью.
3. **Рефакторинг `apps/bot`:** убрать зависимость от Firebase-именований; все вызовы через `BACKEND_URL`.
4. **Группа vs личка:** разделить handlers по `chat.type` / `chat_id`, читать prefs (`telegramGroupChatId`, `telegram_user_id`).
5. **Команды и reply** в группе — после стабильной авторизации и минимального API.
6. **Расширение `emit_domain_event`** на остальные домены — параллельно с продуктом уведомлений.
