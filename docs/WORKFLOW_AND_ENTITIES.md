# Рабочий процесс и сущности (БД ↔ фронт/бэк)

## Как мы работаем: кто что меняет

### 1. Деплой — только изменения кода

- При пуше в `main` деплоятся **только**:
  - код фронта (React/Vite),
  - код бэкенда (FastAPI),
  - код Telegram-бота,
  - конфиги (nginx и т.д.).
- **БД уже есть и с данными.** Данные хранятся в Docker volume `pgdata` и при деплое **не затираются**. Подробно — в [DEPLOY_FLOW.md](DEPLOY_FLOW.md).

### 2. Работа с данными в БД

- **В проде:** работа с данными (просмотр, ручные правки, проверка) — через **админ-панель** в приложении (раздел «Админ-панель»: таблицы, логи, метрики, тесты) или при необходимости напрямую на сервере (pgAdmin, Adminer, `psql`).
- **Мы с тобой** (разработка): меняем **только фронт и бэк**. Схему БД меняем только через **миграции Alembic** (новые таблицы/колонки), чтобы при следующем деплое данные не ломались.
- **Ты на сервере:** при необходимости правишь данные напрямую (SQL, дампы, бэкапы). Деплой по `main` эти данные не трогает.

Итого: **да, верно** — мы сращиваем фронт и бэк через таблицы; БД с данными не пересоздаём при деплое; работа с данными — в админке (и при необходимости у тебя на сервере).

## Как добавлять новую сущность в БД

Когда появляется новая сущность (новая таблица или поля), делаем так:

1. **Бэкенд:** добавляем модель в `apps/api/app/models/` (или в существующий файл), регистрируем в `app/models/__init__.py`.
2. **Миграция:** создаём файл в `apps/api/alembic/versions/`, например `005_weekly_plans.py`. В нём в `upgrade()` — `op.create_table(...)` или `op.add_column(...)`, в `down_revision` указываем предыдущую ревизию.
3. **Применение схемы в БД:**
   - **При деплое:** отдельной командой писать не нужно. При старте бэкенда в `lifespan` вызывается `alembic upgrade head`, поэтому после пуша в `main` и деплоя новые миграции применяются автоматически.
   - **Локально или вручную на сервере:** из папки `apps/api` выполнить:
     ```bash
     export DATABASE_URL="postgresql+asyncpg://taska:taska@localhost:5433/taska"  # или твой URL
     alembic upgrade head
     ```
   Так мы не дропаем данные: миграции только добавляют таблицы/колонки (или меняют по написанной логике).

4. **Роуты и фронт:** добавляем роутер в `app/routers/`, подключаем в `main.py`, на фронте — API-методы и UI.

Итого: **отдельной командой писать не обязательно** — после деплоя миграции применятся при старте бэкенда. Если нужно применить миграции без полного деплоя (например, только поднял бэкенд локально), запускаешь `alembic upgrade head` вручную.

## Сущности и ТЗ: зачем и как

Чтобы не расходиться в том, «что за таблица/сущность и как она ведёт себя», удобно иметь по каждой сущности **короткую ТЗ** (описание + правила). Тогда:

- по каждой сущности есть одно место, куда смотреть;
- при доработках сверяемся с этой ТЗ и при необходимости сначала правим ТЗ, потом код;
- не отходим от договорённостей без явного обновления документа.

### Что считать сущностью

Сущность = то, что отображается в одной таблице БД и имеет свой API (и при необходимости свой блок в админке). Например: пользователь, задача, проект, клиент, сделка, документ, встреча, пост контент-плана, склад, товар, движение, ревизия, воронка продаж, статус, приоритет, настройки уведомлений, правило автоматизации и т.д.

### Где это описывать

- **Общая схема БД (Postgres):** таблицы, колонки, связи — в `docs/tz/database/`. Сейчас там есть [DATABASE_SCHEMA.md](tz/database/DATABASE_SCHEMA.md) (описание из эры Firestore; для Postgres можно завести отдельный файл или обновить этот). Имеет смысл завести актуальный файл под Postgres (например `POSTGRES_SCHEMA.md` или обновить существующий), где перечислены таблицы и поля в виде, соответствующем текущему коду (snake_case, типы из моделей SQLAlchemy).
- **По модулям/сущностям:** в `docs/tz/modules/` уже есть ТЗ по модулям (CRM, задачи, контент, финансы, склад, HR, настройки, уведомления). По каждой сущности можно:
  - либо добавить подраздел в соответствующий модульный файл,
  - либо завести отдельный файл под сущность, например `docs/tz/entities/TASK.md`, `docs/tz/entities/DEAL.md`, и т.д.

В каждом таком описании полезно указать:

- название сущности и соответствующую таблицу БД;
- поля (и при необходимости типы/ограничения);
- связи с другими сущностями (внешние ключи, логические связи);
- правила: кто может создавать/редактировать/удалять, мягкое удаление (is_archived), обязательные поля, уникальность и т.д.;
- как сущность отображается во фронте и в админке (если важно).
