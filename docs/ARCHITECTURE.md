# Архитектура tipa.taska.uz

Документ описывает **как устроено в коде сейчас**, без желаемого будущего состояния. Источник правды — репозиторий (`apps/api`, `apps/web`, `apps/bot`).

## Компоненты


| Часть            | Технологии                                | Роль                                                                         |
| ---------------- | ----------------------------------------- | ---------------------------------------------------------------------------- |
| **Frontend**     | Vite + React, SPA                         | UI; ходит в API по `/api/*` (прокси в dev, nginx в prod).                    |
| **Backend**      | FastAPI, SQLAlchemy async, Alembic        | REST API, вебхуки Meta, фоновые циклы в процессе приложения.                 |
| **PostgreSQL**   | Единственный источник правды по сущностям | Задачи, сделки, клиенты, сообщения, уведомления, пользователи и т.д.         |
| **Redis**        | Stream (`XADD`)                           | Журнал доменных событий; отдельного consumer’а в репозитории **нет**.        |
| **Telegram bot** | `python-telegram-bot`, отдельный процесс  | Опрос API, зеркалирование веб-чата в личку, рассылки в группу по расписанию. |


## Поток данных: доменные события и уведомления

1. Роутеры при изменениях вызывают `emit_domain_event` / `log_entity_mutation` (`apps/api/app/services/domain_events.py`).
2. Событие **пишется в таблицу** `notification_events` (Postgres).
3. Параллельно выполняется `**publish_domain_event`** → Redis Stream (`REDIS_EVENTS_STREAM`, по умолчанию `events.domain.v1`) через `XADD` с ограничением длины (`maxlen≈100000`). Если Redis недоступен, публикация в стрим падает, но запись в БД уже есть (флаги `published_to_stream`, `stream_id`).
4. В том же запросе вызывается `**process_domain_event**` (`notification_hub.py`): по типу события строятся получатели, создаются записи `notifications` и `notification_deliveries`, при включённых каналах — зеркало в **внутренний чат** (`inbox_messages`).

**Важно:** обработка событий **синхронная в рамках HTTP-запроса**, не через отдельный воркер, читающий Redis. Стрим — это **аудит/буфер/мониторинг**, а не очередь задач с `XREADGROUP` в этом репозитории (поиск по коду: нет потребителей стрима).

## «Мгновенность» в интерфейсе


| Канал                                 | Механизм                                                                                                                                                                                                                                                         | Задержка / ограничения                                                                                                                                                                                                             |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **In-app уведомления (колокольчик)**  | После создания уведомления вызывается `**realtime_hub.emit`** — in-memory WebSocket-хаб в процессе API (`notifications_realtime.py`). Клиент: `NotificationCenterContext` подключается к `GET ws` → фактически `WebSocket` на `/api/notifications/ws/{user_id}`. | Пока клиент в онлайне и WS жив — пуш приходит сразу. **Нет общей шины между несколькими инстансами API:** при горизонтальном масштабировании пользователь должен попасть на тот же процесс, где открыт сокет (обычно один воркер). |
| **Nginx / прокси**                    | Если не настроен upgrade для WebSocket, браузер получает ошибку; в коде фронта при ошибке WS отключается на сессию (`sessionStorage`), чтобы не спамить консоль.                                                                                                 | Может выглядеть как «уведомления только после перезагрузки» — проверьте прокси.                                                                                                                                                    |
| **Внутренний чат между сотрудниками** | `MiniMessenger` **каждые 5 секунд** дергает `GET /messages` (inbox/outbox) и мержит в локальное хранилище.                                                                                                                                                       | Это **опрос**, не WebSocket. Задержка до ~5 с — ожидаемое поведение кода, не баг сети.                                                                                                                                             |
| **Telegram (исходящие из системы)**   | Записи `notification_deliveries` со статусом `pending`; раз в **5 секунд** в `main.py` запускается `run_pending_deliveries` (отправка в Telegram/e-mail с бэкоффом).                                                                                             | Не realtime; очередь с интервалом опроса воркера.                                                                                                                                                                                  |
| **Telegram лиды (входящие)**          | **Server-side polling** Telegram `getUpdates` по токену из настроек воронки (`funnel.sources.telegram.botToken`). Состояние offset хранится в Postgres (`telegram_integration_state`). При создании сделки эмитится `deal.assigned`.                            | Задержка = интервал polling (по умолчанию 5с). Токен не хранится/не используется в браузере.                                                                                                                                      |
| **Зеркало веб-чата в Telegram (бот)** | Джобы бота периодически читают сообщения через API и шлют в чат пользователя/группу (см. `apps/bot/taska_bot/jobs/`).                                                                                                                                            | Зависит от интервала job’ов и доступности API.                                                                                                                                                                                     |


Итог: **«мгновенно»** в продукте сейчас реалистично в первую очередь для **колокольника через WebSocket** при корректном прокси. **Чат сотрудник и Telegram-зеркала** по дизайну кода **не** претендуют на миллисекундную доставку.

## Meta (Instagram / Messenger)

- Входящие от Meta: `**POST /webhook/meta`** (без префикса `/api`, см. `main.py`) — верификация `GET`, приём событий `POST`.
- Исходящие через Graph API — токены в окружении (`META_*` в `config.py`).

## База данных

- Схема в `apps/api/app/models/`, миграции Alembic.
- Сделки (`deals`): поля `client_id`, `contact_name` и др. маппятся из JSON в camelCase (`clientId`, `contactName`) в роутере. Расхождения «клиент / контакт в UI и в данных» часто упираются в то, **как фронт заполняет и перезаписывает** эти поля при редактировании — это зона отдельной отладки, не описывается здесь как спецификация.

## Известные архитектурные ограничения (честный список)

Подходит для бэклога и обсуждения приоритетов; не исчерпывающий список багов.

- Внутренний чат — polling, не push.
- Redis Stream не используется как очередь с подписчиками в коде — только запись.
- WebSocket-уведомления привязаны к одному процессу API.
- Часть сценариев в UI (фильтры контент-плана только в одном представлении, странное поведение полей сделки) — **продуктовые/фронтовые** несогласованности; требуют воспроизведения и задач в трекере, а не «починки одной строкой в архитектуре».
