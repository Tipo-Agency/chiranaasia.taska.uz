# Лимиты, одновременные пользователи и надёжность

## Сколько пользователей одновременно?

**Жёсткого лимита в коде нет.** Всё упирается в ресурсы сервера и настройки.

- **Backend (uvicorn):** по умолчанию один воркер. Для 10–50 одновременных пользователей обычно хватает. Для большей нагрузки задайте в запуске несколько воркеров, например:  
  `uvicorn app.main:app --workers 4 --host 0.0.0.0 --port 8000`
- **PostgreSQL:** пул соединений (по умолчанию в SQLAlchemy до ~5 на процесс). При нескольких воркерах суммарное число соединений растёт. При необходимости увеличьте `pool_size` в настройках БД и лимиты в самом Postgres (`max_connections`).
- **Практика:** для типичного внутреннего/малого бизнеса (до десятков активных пользователей) текущей схемы достаточно. Для сотен пользователей нужны нагрузочные тесты и настройка воркеров/пула БД.

Итого: система не ограничивает число пользователей по коду; лимит задаётся конфигурацией сервера и БД.

## Проверки «дошло / не дошло» (сообщения и задачи)

Сейчас **таких проверок нет**:

- Нет статусов доставки сообщений (delivered/read).
- Нет явной «доставки» задач на бэкенд: фронт оптимистично обновляет интерфейс и шлёт запрос; при ошибке API пользователь видит уведомление «Ошибка сохранения...» (после доработок в фронте).

Что можно добавить позже (по желанию):

- Для сообщений: хранение статуса доставки/прочтения в БД и отображение в UI.
- Для задач/сделок: после каждого сохранения проверять ответ API (уже так и есть) и при ошибке показывать сообщение (уже сделано). При необходимости — повторная отправка (retry) с бэкенда или очереди.

## Если что-то «отвалилось»

- **Сеть/бэкенд недоступны:** все вызовы API (сохранение задач, постов, сделок, настроек и т.д.) при ошибке показывают уведомление пользователю («Не удалось сохранить задачу», «Ошибка сохранения поста» и т.п.). Локальное состояние на фронте уже обновлено (оптимистичное обновление), но данные на сервер не ушли — пользователь видит ошибку и может повторить действие (например, сохранить снова).
- **Падение бэкенда:** пока бэкенд не поднят, новые сохранения будут с ошибкой; после поднятия всё снова работает. Данные в БД не теряются из-за падения фронта.
- **Рекомендация:** мониторить `/health` (и при падении — алерты в Telegram, как в HEALTH_LOGS_ALERTS.md).

## Почему не всё «в беке» работало / доработки

- Раньше часть вызовов API шла без обработки ошибок: при падении сети или 500 пользователь не видел сообщения об ошибке и не понимал, что сохранение не прошло.
- **Сделано:** для сохранений (задачи, посты, сделки, клиенты, настройки, таблицы, документы, папки, встречи, финансы, правила и т.д.) добавлена обработка ошибок: при неуспешном ответе API показывается уведомление («Ошибка сохранения...», «Не удалось сохранить задачу» и т.п.). Так и создание, и обновление сущностей либо проходят, либо явно сообщают о сбое.

Если какая-то сущность по-прежнему не создаётся:

- Убедитесь, что бэкенд запущен и доступен (например, открыть `http://localhost:8000/health`).
- Откройте DevTools → Network и посмотрите запрос при создании: статус (4xx/5xx) и тело ответа подскажут причину (валидация, дубликат id, отсутствующее поле и т.д.).

## Кратко

| Вопрос | Ответ |
|--------|--------|
| Лимит пользователей в коде | Нет, зависит от воркеров и БД |
| Проверки доставки сообщений/задач | Нет; при ошибке API пользователь видит уведомление об ошибке сохранения |
| Если отвалился бэкенд/сеть | Показывается ошибка сохранения; после восстановления можно повторить действие |
| Создание сущностей | Все основные создания идут через API; при ошибке показывается сообщение пользователю |
