# Architecture Decision Records + Engineering Principles

Этот документ отвечает на три блока:

1. **Почему мы сделали именно так** — ADR для уже принятых решений
2. **Как принимать решения дальше** — баланс скорости и качества
3. **Единый реестр инженерных правил** — Часть III (с отсылками к `ARCHITECTURE` / `API`)

Новые решения, затрагивающие архитектуру или контракты, фиксируйте ADR в этом файле и при необходимости добавляйте ссылку из [ARCHITECTURE.md](./ARCHITECTURE.md).

## Часть I. Architecture Decision Records (ADR)

ADR — короткая запись о значимом решении: контекст, варианты, выбор, последствия.  
Зачем: через год новый разработчик (или ты сам) не будет гадать «почему это вот так».

### Формат

```markdown
## ADR-XXX: Заголовок решения

**Статус:** принято | устарело | заменено ADR-YYY
**Дата:** YYYY-MM-DD
**Участники:** кто принимал решение

### Контекст
Что происходило, какая была проблема или развилка.

### Рассмотренные варианты
Что ещё рассматривали и почему отклонили.

### Решение
Что выбрали.

### Последствия
Что стало лучше, что хуже, какие обязательства взяли на себя.

### Пересмотреть когда
**Обязательный блок:** конкретный триггер (метрика, дата, событие), при котором решение перечитывают — иначе ADR превращается в «навсегда» без владельца.
```

### ADR-001: Monolith вместо Microservices

**Статус:** принято  
**Дата:** 2025 (начало проекта)

#### Контекст

Продукт объединяет задачи, CRM, финансы, HR, склад. Команда — 1–3 бэкенд-разработчика.

#### Рассмотренные варианты

**Microservices** — отдельный сервис на каждый домен (tasks-service, crm-service, finance-service).  
Плюсы: независимое масштабирование, независимые деплои.  
Минусы: distributed transactions, service discovery, сетевые задержки, operational overhead в 5–10× больше.

**Monolith** — один FastAPI процесс, все домены внутри.  
Плюсы: один деплой, транзакции в рамках одной БД, простота отладки.  
Минусы: общий uptime, надо следить за границами модулей внутри.

#### Решение

Monolith с чёткими внутренними границами (`routers/` → `services/` → `domain/`).  
Фоновые задачи — отдельные воркер-процессы (не сервисы, но и не в API).

#### Последствия

✅ Команда из 1–3 человек может итерировать быстро  
✅ Транзакции между доменами (сделка + уведомление) — без distributed saga  
✅ Один docker-compose файл  
⚠️ При росте команды > 5 бэкендеров — граница модулей начнёт размываться, нужна дисциплина  
⚠️ Нельзя масштабировать только один домен — только весь API

**Пересмотреть когда:** команда > 5 бэкендеров с разными зонами ответственности, или появятся принципиально разные требования к надёжности/latency у разных доменов.

### ADR-002: PostgreSQL как единственный источник правды

**Статус:** принято  
**Дата:** 2025

#### Контекст

Нужно хранить сущности (задачи, сделки, клиенты, уведомления), историю изменений, очереди доставок.  
Соблазн: Redis для кэша всего, отдельные хранилища per-domain.

#### Рассмотренные варианты

**Event Sourcing** — хранить только события, состояние восстанавливать из них.  
Отклонено: сложность проекции, eventual consistency в UI — избыточно при наших требованиях.

**CQRS с отдельным read-store** — PostgreSQL для записи, Redis/Elasticsearch для чтения.  
Отклонено: два хранилища = двойная сложность синхронизации, не обоснована нагрузкой.

**PostgreSQL + Redis только для очередей/кэша** — PostgreSQL SoT, Redis как вспомогательный.

#### Решение

PostgreSQL — единственный источник правды для всех сущностей.  
Redis — очереди задач (Streams), Pub/Sub для WebSocket, short-lived кэш (idempotency keys, offsets).

#### Последствия

✅ Одна транзакция может захватить любой набор сущностей  
✅ Нет проблемы синхронизации между хранилищами  
✅ Alembic — один источник истины по схеме  
⚠️ При > 10M строк в tasks/messages — придётся думать о партиционировании  
⚠️ Redis недоступен → очереди стоят, но данные не теряются (в БД)

**Пересмотреть когда:** таблицы ключевых сущностей стабильно > 10M строк, появились требования к отдельному read-store / репликам только для чтения, или Redis начинает рассматриваться как SoT для бизнес-данных.

### ADR-003: Redis Streams вместо Kafka или RabbitMQ

**Статус:** принято  
**Дата:** 2025

#### Контекст

Нужны очереди для фоновых задач: доставка уведомлений, обработка вебхуков, синхронизация Telegram.  
Объём: < 10K сообщений/день на старте.

#### Рассмотренные варианты

**Kafka** — высокая пропускная способность, долгое хранение, consumer groups, replay.  
Отклонено: отдельный кластер Zookeeper/KRaft, overhead для < 10K сообщений/день неоправдан, operational complexity.

**RabbitMQ** — классическая очередь, хорошая документация.  
Отклонено: ещё один процесс в инфраструктуре, Redis уже есть и умеет то же самое через Streams.

**Redis Streams** — XADD/XREADGROUP, consumer groups, ACK.  
Выбрано: Redis уже в стеке (кэш, Pub/Sub), Streams дают достаточный функционал.

#### Последствия

✅ Один инфраструктурный компонент делает и кэш, и очереди, и Pub/Sub  
✅ Нет дополнительного сервиса  
✅ XREADGROUP + consumer groups = конкурентная обработка  
⚠️ Нет built-in replay истории > maxlen сообщений  
⚠️ При > 1M сообщений/день — Kafka станет обоснованным выбором

**Пересмотреть когда:** объём > 500K сообщений/день или появится требование хранить историю событий > 30 дней для replay.

### ADR-004: Self-hosted VPS вместо Kubernetes/облака

**Статус:** принято  
**Дата:** 2025

#### Контекст

Продукт для одного заказчика. Команда без выделенного DevOps.  
Нужно: простой деплой, контроль над данными, предсказуемая стоимость.

#### Рассмотренные варианты

**AWS EKS / Google GKE** — managed Kubernetes.  
Отклонено: $200–500+/мес для нашей нагрузки, требует K8s-экспертизы, overkill.

**AWS ECS / Fargate** — managed контейнеры без K8s.  
Отклонено: vendor lock-in, переменная стоимость, сложнее дебажить.

**VPS + Docker Compose** — один сервер, всё в контейнерах.  
Выбрано: $20–50/мес, полный контроль, Docker Compose знаком всем.

#### Последствия

✅ Предсказуемая стоимость  
✅ Данные у заказчика, не у Amazon  
✅ docker-compose up — достаточно для деплоя  
⚠️ Нет auto-scaling (не нужно при 200 пользователях)  
⚠️ Single point of failure — нет HA из коробки  
⚠️ Обновление ОС / security patches — ответственность команды

**Пересмотреть когда:** несколько заказчиков на одной платформе (multi-tenant), или требование SLA > 99.5% uptime.

### ADR-005: JWT в HttpOnly Cookies вместо sessionStorage

**Статус:** принято (**as-built:** бэкенд выдаёт HttpOnly `access_token`, фронт не кладёт JWT в `sessionStorage` для API; см. `apps/web/CLAUDE.md`, `apps/api/CLAUDE.md`)  
**Дата:** апрель 2026

#### Контекст

Ранее обсуждался перенос с хранения JWT в `sessionStorage` (уязвимо к XSS: скрипт на странице мог украсть токен) на cookie-модель.

#### Рассмотренные варианты

**sessionStorage** (legacy) — токен доступен JS; XSS → кража сессии.  
Отклонено для основного API-логина.

**localStorage** — то же, но переживает перезагрузку.  
Хуже sessionStorage с точки зрения безопасности.

**HttpOnly Cookie** — браузер шлёт автоматически, JS не имеет доступа.  
Требует CSRF-защиту (Double Submit Cookie паттерн).

#### Решение

HttpOnly cookie `access_token` + обычная cookie `csrf_token` (читается JS) + заголовок `X-CSRF-Token` на мутирующих запросах.

#### Последствия

✅ XSS не может украсть токен  
✅ Браузер шлёт cookie автоматически — меньше кода в api.ts  
⚠️ Требует CSRF middleware на бэкенде  
⚠️ Сложнее тестировать (нужно учитывать cookies в тестах)  
✅ Фронт для сессии API опирается на cookie (см. `apiClient.ts`).

**Пересмотреть когда:** смена модели сессий (opaque server-side sessions, короткоживущий access + refresh-only cookie и т.д.) или появление нового клиента (мобильное приложение) с иным транспортом токена.

### ADR-006: Polling вместо Webhook для Telegram лидов

**Статус:** принято (осознанный компромисс)  
**Дата:** 2025

#### Контекст

Входящие сообщения от лидов в Telegram можно получать двумя способами.

#### Рассмотренные варианты

**Webhook** — Telegram сам пушит обновления на наш URL.  
Плюсы: мгновенно, нет polling-нагрузки.  
Минусы: требует публичный HTTPS URL, сложнее локальная разработка (ngrok или аналог).

**getUpdates polling** — сами опрашиваем Telegram каждые N секунд.  
Плюсы: работает везде, проще в dev, не нужен публичный URL для бота.  
Минусы: задержка = интервал polling, CPU на сетевые вызовы.

#### Решение

Polling с интервалом 5 сек. При < 10 воронках — приемлемая нагрузка.

#### Последствия

✅ Работает в локальной разработке без ngrok  
✅ Один механизм для всех воронок  
⚠️ Задержка доставки ~5 сек (приемлемо для B2B)  
⚠️ При 10 воронках = ~172K запросов/день к Telegram API

**Пересмотреть когда:** > 20 воронок или требование latency < 1 сек.

## Часть II. Velocity vs Architecture — баланс

### Проблема

Каждая команда рано или поздно приходит к спору:  
— *«Давай быстро, потом отрефакторим»*  
— *«Нет, надо сделать правильно сразу»*

Оба правы. Оба не правы. Проблема — в отсутствии явного договора.

### Фреймворк принятия решения

Перед любым «срезанием угла» ответь на три вопроса:

```
1. ОБРАТИМОСТЬ
   Это решение легко изменить потом? (< 2 часов работы)
   → ДА: можно срезать угол
   → НЕТ: думать дважды

2. РАСПРОСТРАНЕНИЕ
   Это паттерн, который скопируют в 10 других местах?
   → НЕТ: локально плохо, терпимо
   → ДА: не срезать, станет стандартом

3. БЕЗОПАСНОСТЬ / ДАННЫЕ
   Это затрагивает данные пользователей, деньги, аутентификацию?
   → ДА: никаких компромиссов, только правильно
   → НЕТ: можно обсуждать
```

### Матрица решений


| Ситуация                                     | Действие                                 |
| -------------------------------------------- | ---------------------------------------- |
| Новая фича, бизнес ждёт, архитектурно спорно | Быстро + создать задачу-долг с дедлайном |
| Исправление бага в prod                      | Быстро + hotfix + потом правильно        |
| Безопасность (XSS, JWT, CSRF)                | Только правильно, без компромиссов       |
| Новый паттерн, который размножится           | Правильно сразу, иначе рефакторинг ×10   |
| Миграция БД с типами (VARCHAR→UUID)          | Итеративно, не «потом всё сразу»         |
| Тесты на новую фичу                          | Минимум happy path + один error case     |


### Технический долг — явный бюджет

Технический долг — не ошибка, это инструмент. Как кредит: полезен когда осознан, разрушителен когда накапливается молча.

**Правила:**

```
✅ Каждый сознательный долг — задача в трекере с меткой "tech-debt"
   Формат: "Сделано быстро: [что]. Правильно: [как]. Цена: [сложность]"

✅ Tech-debt задачи — в каждом спринте не менее 20% capacity
   Иначе долг только накапливается

✅ Долг, который блокирует новые фичи → P1, берём в следующий спринт

❌ "Потом отрефакторим" без задачи в трекере = никогда
❌ Архитектурные компромиссы в безопасности и данных
❌ "Пока сойдёт" для паттернов, которые размножатся
```

**Временные реализации:** осознанный «костыль» или упрощение оформляется задачей с **явным дедлайном** (дата или конкретный спринт). Без срока — не временное решение, а долг без владельца.

### Как коммуницировать с бизнесом

Бизнес не против качества — он против непонятных задержек.  
Техническая команда должна уметь объяснить цену в понятных терминах:

```
❌ "Нам нужно рефакторить архитектуру"
   (непонятно, страшно, кажется что теряем время)

✅ "Сейчас добавление нового поля в сделку занимает 4 часа.
   После рефакторинга схем — 30 минут. Вложим 1 день, сэкономим
   5+ дней на следующих 10 фичах."

❌ "Мы накопили технический долг"

✅ "У нас 3 места в коде без валидации входных данных.
   Это риск потери или порчи данных. Закроем за 2 дня."
```

### Velocity Indicators (сигналы что архитектура тормозит)

Если замечаешь эти симптомы — время остановиться и заплатить долг:

```
🔴 Новая фича требует изменений в > 5 файлах, которые «не должны были меняться»
🔴 Разработчик боится трогать модуль (FinanceView.tsx, useAppLogic.ts)
🔴 Один и тот же баг всплывает в разных местах (нет единого источника правды)
🔴 Code review занимает > 2 дней — значит изменение слишком большое
🔴 «Давай не будем тесты писать, некогда» — системный паттерн, не разовый
🔴 Деплой вызывает страх (нет тестов, нет rollback-плана)
```

### Когда «правильно сразу» — всегда

Есть категории, где компромисс недопустим независимо от давления:

```
🔒 БЕЗОПАСНОСТЬ
   XSS, SQL injection, хранение токенов, CSRF — только правильно.
   Пример: "давай пока без DOMPurify, потом добавим" → нет.

💾 ЦЕЛОСТНОСТЬ ДАННЫХ
   Транзакции при мульти-таблица операциях — всегда.
   Потеря данных клиента не компенсируется скоростью деплоя.

📋 АУДИТ
   Мутации финансовых данных без audit_log — нет.
   Регулятор или заказчик спросит — нужна история.

🔑 СЕКРЕТЫ
   SECRET_KEY без дефолта, токены зашифрованы — всегда.
   "Потом уберём захардкоженный токен" — классика утечки данных.
```

### ADR для будущих решений

Когда нужно писать ADR:

- Выбор между двумя принципиально разными подходами
- Решение, которое будет дорого менять (схема БД, API контракт)
- Изменение схемы БД, затрагивающее **контракт API**, **типы для клиентов** или **семантику данных** (см. Часть III)
- Осознанный отказ от "правильного" решения в пользу скорости
- Отказ от популярного инструмента (и почему)

Когда ADR не нужен:

- Выбор библиотеки (DOMPurify vs sanitize-html) — просто сделай
- Рефакторинг без изменения поведения
- Именование переменных

**Шаблон для быстрого ADR (5 минут):**

```markdown
## ADR-00X: [Заголовок]

**Контекст:** [одно предложение — что происходило]
**Варианты:** [перечислить через запятую]
**Выбор:** [что выбрали]
**Почему:** [2-3 предложения]
**Цена:** [что стало сложнее или какой долг взяли]
**Пересмотреть когда:** [триггер]
```

## Часть III. Инженерные правила (реестр)

Краткие договорённости команды; подробности и примеры — в `docs/ARCHITECTURE.md`, `docs/API.md`, `docs/DATABASE.md`, `docs/SECURITY.md`.

### Очереди и имена потоков

- Рабочие очереди (Redis Streams, фоновые джобы): `**queue.<домен>.vN**` (например `queue.notifications.v1`, `queue.integrations.v1`). Смена формата сообщения → новый поток `**…v2**`, старый дренируется и выключается, чтобы старые воркеры не ломались на новом payload.
- Публикация доменных/внешних событий (если отдельный канал): `**events.<домен>.vN**` — тот же принцип версии в суффиксе.

### Семантика доставки

- Для **очередей, вебхуков, фоновой обработки и повторов** действует **at-least-once**: повтор доставки возможен. **Exactly-once не обещаем.** Обработчики и побочные эффекты — **идемпотентны** (см. `ARCHITECTURE.md` §7, §15).

### DLQ

- Политика хранения (например **7 дней** в `dead_letter_queue`), **requeue** в исходный версионированный stream после исправления — см. `ARCHITECTURE.md` §6.4.
- DLQ без **админ-интерфейса или операционного скрипта** (просмотр, фильтр, безопасный requeue) не считается завершённым контрактом — иначе инциденты не разбираются.

### WebSocket

- **Лимит соединений на пользователя** (целевое значение, напр. max **10**) — `ARCHITECTURE.md` §5.
- **Истечение cookie/JWT:** соединение **закрывается** сервером; клиент обязан **обновить сессию** и **переподключиться** — `API.md` §11.
- Доставка уведомлений по WS — **best-effort**; полнота и история — **БД + REST**, не сокет.

### Cookies (production)

- Для auth-cookies: `**Secure`**, `**HttpOnly**` (где уместно), `**SameSite=Lax**` или `**Strict**` (по сценарию) — зафиксировано в `ARCHITECTURE.md` §8 и `SECURITY.md`; фактические флаги сверять с кодом.

### [CURRENT] / [TARGET]

- В архитектурных документах явно разделяем фактическое и целевое поведение; **сводная таблица** — `ARCHITECTURE.md` §13.
- **Доменный hub (`process_domain_event`):** в HTTP при `DOMAIN_EVENTS_HUB_ASYNC=false` или если событие не опубликовано в Redis; во **`domain_events_worker`** при `DOMAIN_EVENTS_HUB_ASYNC=true` и успешном XADD — **[CURRENT]** (см. `QUEUES.md`, `ARCHITECTURE.md` §6–§7).
- **WebSocket / realtime:** доставка через **Redis Pub/Sub** между инстансами API + локальные WS — **[CURRENT]**; жёсткий лимит вкладок на пользователя — см. `ARCHITECTURE.md` §5.
- **Фоновые задачи:** отдельные процессы (`notifications_worker`, `integrations_worker`, `domain_events_worker`, `retention_worker`) и **Redis Streams** — **[CURRENT]**; остаточный asyncio в Uvicorn без новых тяжёлых циклов — не расширять без ADR.

### API, пагинация, лимиты, bulk

- **Idempotency-Key** в хранилище: `**idempotency:{METHOD}:{normalized_path}:{key}`**, не голый `{key}` — `API.md` §3.
- **Cursor:** поле времени в курсоре **immutable**; сортировка ленты фиксирована `**ORDER BY created_at DESC, id DESC`** — `API.md` §4.2.
- **Rate limit key:** авторизованный запрос → `**user_id`**; иначе → **IP** (из доверенного `X-Forwarded-For`) — `API.md` §6.
- **Bulk:** по умолчанию **partial success** с явным `failed[]` / `skipped`; атомарность «всё или ничего» только у отдельных, явно помеченных эндпоинтов — `API.md` §10.
- **ETag:** каноническая сериализация через `**json.dumps(..., sort_keys=True, …)`**, не `str(dict)` — `API.md` §9.
- **Enum-строки в API:** новые значения **можно добавлять**; существующие **не удалять и не переименовывать** без мажорной версии/миграции клиентов — `API.md` §0.
- **Deprecation:** снятие эндпоинта не раньше чем через **3 месяца** после анонса — `API.md` §0.

### Вебхуки

- Кроме подписи — защита от **replay** (timestamp / окно свежести + идемпотентность по id события) — `API.md` §8.

### Доменные события и порядок

- **Глобальный порядок между разными consumer'ами не гарантируется** (масштабирование, несколько воркеров). Проектировать обработчики без предположения «событие A всегда придёт раньше B» между разными типами/очередями, если нет явного упорядочивания в одном потоке.

### Ретраи

- До **5** попыток, backoff **1m → 5m → 15m → 60m**, затем DLQ — `ARCHITECTURE.md` §6.3–6.4.

### Наблюдаемость

- Каждая **структурированная** запись лога в цепочке обработки HTTP-запроса содержит `**request_id`** (обязательное поле). Если клиент не прислал `X-Request-ID`, сервер генерирует — `API.md` §12.

### Схема БД и ADR

- Изменение схемы PostgreSQL, которое затрагивает **контракт API**, **типы для клиентов** или **семантику данных**, сопровождается **новым или обновлённым ADR** и миграцией по правилам `DATABASE.md`.
