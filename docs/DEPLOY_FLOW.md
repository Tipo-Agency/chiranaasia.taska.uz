# Как работает автодеплой (пошагово)

## Триггер

- **Когда:** при любом `git push` в ветку `main`.
- **Как запустить деплой:** из корня репозитория выполнить `npm run push` (или `npm run deploy`) — это сделает `git push origin main`, после чего GitHub Actions сам выполнит деплой на сервер.
- **Кто:** GitHub Actions запускает workflow из `.github/workflows/deploy.yml`.
- **Что делает workflow:** подключается по SSH к серверу и выполняет один скрипт — `ops/scripts/deploy.sh`, передавая в него переменные из GitHub Secrets.

Передаваемые переменные (все опциональны, кроме указанных в README):

| Переменная | Описание |
|------------|----------|
| `SERVER_PATH` | Путь к репо на сервере (например `/var/www/tipa.taska.uz`) |
| `TELEGRAM_BOT_TOKEN` | Токен бота для деплоя бота |
| `BACKEND_URL` | URL API для бота (например `http://127.0.0.1:8003`) |
| `NGINX_SITE_NAME` | Имя сайта nginx (по умолчанию `tipa.taska.uz`) |

---

## Что делает `deploy.sh` (по шагам)

### Step 1: Права на файлы

- `chown` и `chmod` на `$SERVER_PATH` и `.git`, чтобы пользователь деплоя мог читать/писать репозиторий.
- Нужно, чтобы после `git`-операций не было проблем с правами.

---

### Step 2: Обновление кода (только по изменениям)

- `git fetch origin` и `git merge origin/main --ff-only` — код обновляется только по изменениям, без жёсткой очистки и без удаления неотслеживаемых файлов.
- **Нет `git reset --hard` и `git clean`** — локальные изменения и неотслеживаемые файлы на сервере не затираются. Если merge не fast-forward, деплой завершится с ошибкой (нужен ручной merge на сервере или перезапуск после pull).
- **БД на этом шаге не трогается.** Docker volumes (в т.ч. `pgdata`) сохраняются.

---

### Step 2b: Docker — Postgres и backend

- Выполняется: `docker compose up -d --build db backend`.
- **db:** контейнер Postgres. Данные хранятся в **Docker volume `pgdata`**. При каждом деплое контейнер может пересоздаваться, но том `pgdata` сохраняется — **данные БД не теряются**.
- **backend:** образ пересобирается из `apps/api`, контейнер перезапускается. При старте приложения в `main.py` в `lifespan` вызывается **Alembic** — выполняются миграции до `head`. То есть при деплое применяются только **новые** миграции (добавление таблиц/колонок и т.д.), без полного пересоздания БД.
- Итог: деплой обновляет код бэкенда и схему БД по миграциям, но **не затирает** существующие данные в `pgdata`.

---

### Step 3: Фронтенд

- `npm ci` — чистая установка зависимостей по `package-lock.json`.
- `npm run build:web` — сборка фронта (Vite) в `apps/web/dist`.
- `rsync -a --delete apps/web/dist/ /var/www/frontend/` — содержимое `dist` копируется в `/var/www/frontend`, старое содержимое каталога затирается.
- Nginx раздаёт статику из `/var/www/frontend`. То есть обновляется **только статика фронта**, БД снова не затрагивается.

---

### Step 4: Telegram-бот

- Обновляется `.env` в `apps/bot` (токен, `BACKEND_URL`).
- Копируется unit `telegram-bot.service` в systemd, выполняется `systemctl daemon-reload` и `systemctl restart telegram-bot`.
- Работа с данными — только через API бэкенда; БД деплой не трогает.

---

### Step 5: Nginx

- Копируется `ops/nginx/nginx.conf` в `/etc/nginx/sites-available/$NGINX_SITE_NAME`, делается симлинк в `sites-enabled`, выполняется `nginx -t` и `systemctl reload nginx`.
- Меняется только конфиг и статика, которую отдаёт nginx; БД не затрагивается.

---

## Кратко: что деплоится и что сохраняется

| Что | Деплой | Данные |
|-----|--------|--------|
| Код репо (фронт, бэкенд, бот, скрипты) | Да, перезаписывается из `main` | — |
| Docker-образ backend | Да, пересборка и перезапуск | — |
| Docker-контейнер Postgres | Может перезапускаться | **Данные в volume `pgdata` сохраняются** |
| Схема БД (таблицы, колонки) | Применяются только новые миграции Alembic | Существующие строки не удаляются миграциями по умолчанию |
| Статика фронта | Да, папка `/var/www/frontend` обновляется из `dist` | — |
| Данные в БД (задачи, пользователи, сделки и т.д.) | **Нет** | **Не трогаются** деплоем |

Итого: **деплоим только изменения кода (фронт + бэк + бот) и схемы (миграции). Данные в БД не теряются** — они в Docker volume и не перезаписываются при обновлении репозитория или образов.
