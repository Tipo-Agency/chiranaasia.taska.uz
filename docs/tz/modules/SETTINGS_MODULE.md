# Техническое задание: Модуль настроек (`Settings`)

## 1. Назначение и место в системе

### 1.1. Общая роль
Модуль настроек — это **единый центр администрирования системы**:

- управляет **справочниками и конфигурацией** всех ключевых модулей (Задачи, CRM, Финансы, HR, Бизнес‑процессы, Склад, Контент, Документы);
- управляет **пользователями и доступами**;
- настраивает **уведомления и роботов** (автоматизацию);
- содержит **архив** для всех сущностей (мягкое удаление / восстановление).

Работает только для пользователей с ролью **`ADMIN`**.

### 1.2. Где живёт в UI

- Раздел **«Настройки»**:
  - пункт в сайдбаре снизу (шестерёнка),
  - пункт в дропдауне пользователя (кнопка «Настройки»).
- При открытии показывается страница `apps/web/components/SettingsView.tsx` со следующей структурой:
  - слева — **боковое меню вкладок** (личные / система / задачи / автоматизация / архив),
  - справа — **контент‑панель** выбранной вкладки.

### 1.3. Главные блоки настроек (UI)

1. **ЛИЧНЫЕ**
   - `Профиль` — данные текущего пользователя (имя, логин, телефон, Telegram, аватар, пароль).

2. **СИСТЕМА**
   - `Пользователи` — управление сотрудниками и их ролями.
   - `Проекты` — логические модули / направления (CRM, Склад, Маркетинг и т.п.).
   - `Подразделения` — департаменты компании.
   - `Статьи расходов` — справочник финансовых категорий.
   - `Фонды` — фонды компании (зарплаты, закупки, резерв и т.п.).
   - `Воронки продаж` — конфигурация CRM‑воронок.

3. **ЗАДАЧИ**
   - `Статусы` — справочник статусов задач.
   - `Приоритеты` — справочник приоритетов задач.

4. **АВТОМАТИЗАЦИЯ**
   - `Уведомления и роботы` — настройки Telegram‑уведомлений + правила автоматизации.

5. **РАЗНОЕ**
   - `Архив` — просмотр и восстановление **любой** архивной сущности (задачи, пользователи, сделки, клиенты, документы, встречи и т.д.).

> Внутри каждой вкладки есть краткий текст‑описание, надписи на кнопках и подсказки, чтобы админ понимал, **что именно он сейчас настраивает**.

## 2. Таблицы / страницы (`tables`)

### 2.1. Создание таблицы

#### 2.1.1. Обязательные поля
- **name** (string) - Название таблицы
- **type** (string) - Тип таблицы

#### 2.1.2. Типы таблиц
- **tasks** - Таблица задач
- **docs** - Таблица документов
- **meetings** - Таблица встреч
- **content-plan** - Контент-план
- **backlog** - Беклог (идеи)
- **functionality** - Функционал

#### 2.1.3. Опциональные поля
- **isSystem** (boolean) - Системная таблица (нельзя удалить)

### 2.2. Управление таблицами
- Создание таблицы
- Редактирование таблицы
- Удаление таблицы (кроме системных)
- Мягкое удаление (архив)
- Восстановление из архива

## 3. Уведомления и роботы (`notification_prefs` + `automation_rules`)

### 3.1. Настройки уведомлений (UI «Уведомления»)

#### 3.1.1. Типы уведомлений
- **newTask** - Новая задача
- **statusChange** - Изменение статуса
- **taskAssigned** - Назначение задачи
- **taskComment** - Комментарий к задаче
- **taskDeadline** - Приближение дедлайна
- **dealCreated** - Новая сделка
- **dealStatusChanged** - Изменение стадии сделки
- **clientCreated** - Новый клиент
- **contractCreated** - Новый договор
- **docCreated** - Новый документ
- **meetingCreated** - Новая встреча
- **purchaseRequestCreated** - Новая заявка на покупку

#### 3.1.2. Настройки для каждого типа
```typescript
{
  telegramPersonal: boolean;  // Уведомления в личный чат
  telegramGroup: boolean;     // Уведомления в группу
}
```

### 3.2. Глобальные настройки
- **telegramGroupChatId** (string) - ID группового чата для уведомлений
- **defaultFunnelId** (string) - ID основной воронки для лидов

## 4. Автоматизация (правила `automation_rules`)

### 4.1. Создание правила (UI «Роботы»)

#### 4.1.1. Обязательные поля
- **name** (string) - Название правила
- **isActive** (boolean) - Активно ли правило
- **module** (string) - Модуль: 'tasks' | 'docs' | 'meetings' | 'content' | 'finance' | 'crm' | 'employees' | 'bpm'
- **trigger** (string) - Тип триггера (TriggerType)
- **action** (object) - Действие

#### 4.1.2. Типы триггеров
- **task_created** - Создание задачи
- **task_status_changed** - Изменение статуса задачи
- **task_assigned** - Назначение задачи
- **task_comment** - Комментарий к задаче
- **task_deadline** - Приближение дедлайна
- **deal_created** - Создание сделки
- **deal_status_changed** - Изменение стадии сделки
- И другие...

#### 4.1.3. Условия (conditions)
```typescript
{
  moduleId?: string;      // ID модуля
  statusTo?: string;     // Статус для триггера изменения статуса
  statusFrom?: string;   // Исходный статус
  priority?: string;     // Приоритет
  departmentId?: string; // ID отдела
  categoryId?: string;   // ID категории
}
```

#### 4.1.4. Действия (action)
```typescript
{
  type: 'telegram_message' | 'approval_request' | 'assign_task' | 'change_status';
  template?: string;              // Шаблон сообщения
  targetUser: 'assignee' | 'creator' | 'admin' | 'specific' | 'manager';
  specificUserId?: string;         // ID конкретного пользователя
  buttons?: TelegramButton[];      // Кнопки для Telegram
  approvalType?: string;           // Тип согласования
  approvalEntityId?: string;       // ID сущности для согласования
}
```

### 4.2. Управление правилами
- Создание правила
- Редактирование правила
- Активация/деактивация правила
- Удаление правила

## 6. Telegram‑боты и интеграции

### 6.1. Принцип безопасности

- Все токены ботов и chatId **никогда не хранятся в браузере**.
- Реальные значения задаются только:
  - в `.env` backend’а (`TELEGRAM_BOT_TOKEN_*` и т.д.),
  - либо через защищённые серверные скрипты (`ops/scripts/*`).
- В UI раздел «Интеграции» может:
  - отображать **состояние интеграций** (подключено / не подключено),
  - показывать инструкцию, *куда и что положить в `.env`*.

### 6.2. Связь с backend

- Настройки уведомлений / роботов **не** хранят токены.
- Все HTTP‑запросы в Telegram идут только через backend (см. модуль `apps/api/app/integrations/*`), а не напрямую из фронта.

## 7. Логи активности (`activity`)

### 7.1. Логи активности
- Все действия пользователей логируются
- Отображаются в разделе "Уведомления"

## 8. Технические детали (frontend + backend)

### 8.1. Основные React‑компоненты

- `SettingsView` — корневая страница настроек (`apps/web/components/SettingsView.tsx`):
  - рисует левое меню и подставляет нужный под‑компонент в зависимости от `activeTab`;
  - отвечает за маршрутизацию вкладок и передачу данных/экшенов из `useAppLogic`.
- `ProfileSettings` — профиль пользователя и управление сотрудниками.
- `SpaceSettings` —:
  - под‑вкладка `pages` — настройка таблиц / страниц,
  - под‑вкладка `projects` — настройка модулей,
  - под‑вкладки `statuses` / `priorities` — справочники задач.
- `DepartmentsView` — управление подразделениями.
- `FinanceCategoriesSettings` — статьи расходов.
- `FundsSettings` — фонды.
- `SalesFunnelsSettings` — CRM‑воронки.
- `AutomationSettings` — уведомления + правила автоматизации.
- `ArchiveView` — архив по всем модулям.

### 8.2. Ключевые API‑эндпойнты

- Таблицы:  
  - `GET /api/tables` — получить все таблицы;  
  - `PUT /api/tables` — массовое обновление (создание/редактирование/архив).
- Уведомления:  
  - `GET /api/notification-prefs` — получить настройки уведомлений;  
  - `PUT /api/notification-prefs` — сохранить настройки.
- Правила автоматизации:  
  - `GET /api/automation/rules` — список правил;  
  - `PUT /api/automation/rules` — сохранить весь список.
- Пользователи:  
  - `GET /api/auth/users` — получить пользователей;  
  - `PUT /api/auth/users` — массовое обновление (создание/архив, смена пароля).
- Остальное (подразделения, финансы, воронки и т.д.) — см. соответствующие ТЗ‑модули (`FINANCE_MODULE_TZ.md`, `CRM_MODULE_TZ.md`, `HR_MODULE_TZ.md`).

### 8.3. Таблицы в базе данных

- `tables` — страницы/таблицы.
- `notification_prefs` — JSON‑схема настроек уведомлений.
- `automation_rules` — правила автоматизации.
- `users` — пользователи системы.
- `activity` — логи активности.
- `departments`, `finance_categories`, `funds`, `sales_funnels` и др. — справочники модулей (см. `DATABASE_SCHEMA.md`).
