# Продуктовые модули

Каждый пронумерованный раздел описывает модуль продукта:

- **Что делает** — пользовательская ценность  
- **Компоненты** — где лежит код  
- **API-роуты** — что использует (при необходимости)  
- **Требования** — что должно работать  
- **Приёмочные критерии** — чеклист для приёмки

Ниже сначала — **карта уровней и правил** (архитектура модульности), затем модули по порядку: **Home → Spaces → домены → система**.

---

## Уровни архитектуры модулей

### Продуктовая метафора

Платформа — **операционная система бизнеса**: не набор разрозненных экранов, а слои поверх общих сущностей и API.


| Слой        | Смысл                                                         |
| ----------- | ------------------------------------------------------------- |
| **CRM**     | Продажи и коммуникации с лидами                               |
| **Tasks**   | Исполнение и планирование работ                               |
| **Finance** | Деньги, заявки, дебиторка                                     |
| **HR**      | Люди и оргструктура                                           |
| **BPM**     | Регламенты и экземпляры процессов                             |
| **Spaces**  | Кастомизация: проекты, типы таблиц, «магазин готовых решений» |
| **Home**    | Сводка и быстрые переходы, без собственной бизнес-логики      |


### 1. Core (сквозные, не отдельные «продуктовые» главы в этом файле)

**Auth, Users, Permissions** — реализуются роутерами `auth` / `users`, middleware, JWT/RBAC и экранами в **Настройках**. Отдельного раздела «модуль Auth» здесь нет: это фундамент всех модулей.

### 2. Domain modules (бизнес-домены)

- **CRM** — сделки, клиенты, диалоги (единый модуль, см. §4)  
- **Tasks** — задачи как сущность и глобальный трекер (§3)  
- **Finance**, **HR**, **BPM** — §5–§7

### 3. Platform layer

- **Spaces** — универсальный движок **проектов → таблиц → строк сущностей** по `table.type` (§2)

### 4. Aggregation layer

- **Home** — дашборд-оркестрация: виджеты, агрегаты, deep-link в домены (§1)

### 5. System modules

- **Уведомления** — §8  
- **Настройки** (в т.ч. интеграции, воронки, пользователи) — §9

### Single Source of Truth (владение сущностями)


| Сущность                         | Владеет модуль         | Другие модули                             |
| -------------------------------- | ---------------------- | ----------------------------------------- |
| Deal, Client, InboxMessage (лид) | **CRM**                | только читают/ссылаются по FK             |
| Task                             | **Tasks**              | CRM может ссылаться на задачу             |
| Employee, Department             | **HR**                 |                                           |
| FinanceRequest, BDR, AR          | **Finance**            |                                           |
| BusinessProcess, BPInstance      | **BPM**                |                                           |
| Table, Project (space)           | **Spaces** (контейнер) | строки таблиц — доменные сущности по типу |


### Связи между модулями (cross-module)

- **Task** → опциональная ссылка на **Deal** (и др. при необходимости).  
- **Meeting** → опционально **Deal** / **Client**.  
- **Notification** → полиморфная ссылка `entity_type` + `entity_id` на любую сущность.  
- **Doc** (и вложения) → могут быть привязаны к сущностям (сделка, задача и т.д.) по правилам продукта.

Связи только через **API и общие идентификаторы**, не через импорт чужой бизнес-логики во фронте/сервисах.

### Границы модулей (код и API)

- У каждого домена — **свои префиксы API** (`/api/deals`, `/api/tasks`, …).  
- **Запрещено:** прямой импорт доменной логики модуля A в модуль B (например CRM-хуки внутри Tasks feature и наоборот). Допустимы: вызов **API**, **общие UI-примитивы**, **shared types/utils** без бизнес-правил.  
- На бэкенде — те же границы: `services/` по доменам, без циклических «позвать deals из tasks напрямую»; при необходимости — **application service** или событие.

### Anti-patterns

- **«Каждая страница = модуль»** — неверно. **Модуль = домен (или платформенный слой)**, а не экран.  
- Дублировать **одни и те же фильтры/поиск/assignee** в разных углах одного домена без общего компонента/хука — технический долг (для CRM см. §4).

### Масштабирование продукта

- Новый `**table.type`** в Spaces → **не** новый верхнеуровневый модуль, а **расширение движка Spaces** + экран строки/колонок.  
- Новая **бизнес-сущность с жизненным циклом** → новый **доменный модуль** (или явное расширение существующего) + ADR при необходимости.

### Roadmap-заметка (product / platform)

**Spaces** со временем может стать центральной историей продукта (**platform-as-a-system**): CRM, Tasks, контент и т.д. — **предустановленные конфигурации** таблиц и интеграций, а не обязательно отдельные «коробки». Текущий as-built может отставать от этой формулировки — маркер **[TARGET]** для эволюции UX и навигации.

### UI-структура (согласовано с [FRONTEND.md](./FRONTEND.md))


| Слой        | Роль                                                         |
| ----------- | ------------------------------------------------------------ |
| **Module**  | Данные, стейт модуля, оркестрация, вызовы API/store          |
| **View**    | Отображение по props, без fetch                              |
| **Feature** | Переиспользуемый доменный блок (карточка сделки, фильтр CRM) |
| **UI**      | Примитивы (`components/ui/`)                                 |

## 1. Home (`home`) — слой агрегации

### Что делает

**Рабочий стол** — первый экран после входа. Это **не отдельный бизнес-модуль**: у Home **нет собственных сущностей и правил жизненного цикла**. Он **агрегирует** данные из **Tasks, CRM, Spaces (встречи), Notifications** и ведёт в соответствующие домены по клику.

### Компоненты

```
components/pages/WorkdeskView.tsx        — главная страница
components/features/home/                — виджеты
  ├── TasksSummary.tsx                   — задачи пользователя (домен Tasks)
  ├── DealsSummary.tsx                   — сделки (домен CRM)
  ├── CalendarWidget.tsx                 — встречи (данные из Meetings / таблицы type=meetings)
  └── NotificationsWidget.tsx            — краткая лента / счётчик [TARGET]
```

### Требования

- **Tasks widget:** невыполненные задачи текущего пользователя, ближайшие по дедлайну; переход в TaskModal / задачи.  
- **Deals widget:** сделки assignee = я, stage ∉ {won, lost}; скрывать целиком без `crm.view`, **не** запрашивать API.  
- **Meetings widget:** ближайшие встречи (например 7 дней) — из домена встреч / таблиц `meetings`; secondary по отношению к полноэкранному календарю в Spaces.  
- **Notifications widget [TARGET]:** последние N или счётчик непрочитанных; источник правды — модуль уведомлений + WS.  
- Быстрое создание задачи с рабочего стола (вызов API Tasks).

### Приёмочные критерии

- Загрузка дашборда < 2 с (lazy по виджетам).  
- Без прав CRM блок сделок отсутствует и запросы не уходят.  
- Клик по задаче/сделке открывает соответствующий модальный/маршрут домена.

## 2. Spaces — платформенный модуль

### Что делает

**Spaces** — **ядро кастомизации**: иерархия **Project → Table → строки** сущностей. Один **универсальный движок** рендерит разные **типы таблиц** (`table.type`).  

**Контент-план, календарь встреч, документы** — **не отдельные верхнеуровневые модули продукта** в смысле домена; это **типы таблиц** и связанные экраны внутри Spaces (и виджеты на Home).

**Метафора «магазин решений»:** каждый `type` — готовое бизнес-решение:


| `table.type`    | Продуктовый смысл                               |
| --------------- | ----------------------------------------------- |
| `tasks`         | Таск-трекер внутри проекта                      |
| `backlog`       | Беклог                                          |
| `functionality` | Функциональность                                |
| `content-plan`  | SMM / контент-планер                            |
| `meetings`      | Календарь и встречи (**primary** для календаря) |
| `docs`          | База знаний / документы (**primary** для доков) |
| `aggregate`     | Агрегатор задач из нескольких проектов          |


### Компоненты

```
components/pages/SpacesTabsView.tsx      — выбор пространства
components/modules/SpaceModule.tsx       — активная таблица
components/ContentPlanView.tsx           — UI строк для type=content-plan
components/MeetingsView.tsx              — UI для type=meetings (+ календарная сетка)
components/DocumentsView.tsx           — дерево/список для type=docs
components/DocEditor.tsx                 — редактор (DOMPurify обязателен)
```

### Требования (движок)

- Переключение проектов и таблиц, сохранение `activeTableId` / URL [TARGET].  
- Общие паттерны: шапка таблицы, фильтры по типу сущности, права на уровне проекта/таблицы.

### Тип `content-plan` (ранее отдельный «модуль Контент-план»)

- Таблица постов: платформа, дата, статус, исполнитель.  
- Статусы: draft → ready → scheduled → published / cancelled.  
- Фильтры: платформа, исполнитель, дата.  
- Публичный просмотр: `GET /api/tables/{id}/public` — только `scheduled` / `published` при флаге публичности таблицы.

### Тип `meetings` (календарь — не отдельный модуль верхнего уровня)

- CRUD встреч: название, время, место, участники.  
- Привязка к **Deal** / **Client** (cross-module).  
- Виды: список + календарная сетка (месяц/неделя).  
- Напоминание за 15 мин (если включено) — через уведомления.  
- **Secondary:** виджет на Home (§1).

### Тип `docs` (документы — не отдельный модуль верхнего уровня)

- Rich-text, папки, вложенность.  
- Публичная ссылка при `is_public`.  
- **DOMPurify** на сохранение и рендер.  
- **Secondary:** быстрый доступ / последние документы на Home [TARGET].

### Приёмочные критерии

- Публичный контент-план и публичный документ — только при явных флагах таблицы/документа.  
- `innerHTML` только через `DOMPurify.sanitize(...)`.  
- Встреча с `deal_id` / `client_id` отображается в CRM/клиенте при переходе по ссылке.

## 3. Задачи (`tasks`, `table`)

### Что делает

Домен **задач**: трекер с таблицей, канбаном, гантом, комментариями, вложениями. Экземпляры задач в проектах живут в таблицах типа `**tasks`** (и родственных) внутри **Spaces** (§2); глобальный список — отдельный экран `TasksPage`.

### Компоненты

```
components/pages/TasksPage.tsx           — глобальный список задач
components/TableView.tsx
components/KanbanBoard.tsx
components/GanttView.tsx                 — lazy
components/TaskModal.tsx
components/features/tasks/
```

### Требования

- CRUD полей: title, description, status, priority, assignee, due_date, tags.  
- Опциональная связь с **Deal** (и др.) — поле/ссылка по продукту.  
- Смена статуса из таблицы, канбана, модалки; **PATCH batch** при drag-and-drop.  
- Фильтры и поиск — на сервере.  
- Комментарии — отдельная сущность; вложения — лимиты по [ENTITIES.md](./ENTITIES.md).  
- Архивация вместо удаления.

### Приёмочные критерии

- POST /api/tasks → 201, UI обновляется.  
- Переходы статусов валидируются на сервере.  
- Пагинация ~50, права `tasks.edit` на мутации.  
- Комментарий > лимита — 422.

## 4. CRM — единый модуль (`sales-funnel`, клиенты, диалоги)

### Что делает

Один продуктовый модуль **CRM** объединяет то, что в UI может идти **табами** (воронка, диалоги, клиенты). Это **единая система продаж**, а не три независимых продукта.

**Воронка** — это **представление (view) над сделками** (доска по стадиям), а не отдельный модуль с собственной сущностью «воронка как бизнес-объект» поверх Deal.

### Поддомены


| Поддомен     | Сущность                | Роль                                                                     |
| ------------ | ----------------------- | ------------------------------------------------------------------------ |
| **Deals**    | Deal                    | **Центральная** сущность CRM: стадия, сумма, воронка, assignee, источник |
| **Clients**  | Client                  | **Справочник** контрагентов; сделки ссылаются на клиента                 |
| **Messages** | InboxMessage (и каналы) | **Коммуникационный слой вокруг сделки** (и воронки)                      |


### Единые паттерны CRM [TARGET]

Фильтры, поиск, назначение ответственного и согласованные пустые состояния — **общие** для вкладок воронки, списков сделок, клиентов и диалогов (общий хук/контекст CRM или дизайн-система фильтров), без копипасты трёх независимых реализаций.

### Компоненты

```
components/modules/CRMHubModule.tsx      — хаб: воронка / диалоги / клиенты
components/SalesFunnelView.tsx
  ├── FunnelBoard.tsx
  ├── FunnelCard.tsx
  └── FunnelFilters.tsx
components/modules/CRMModule.tsx
components/pages/ClientChatsPage.tsx     — диалоги (полноэкранный)
components/pages/ClientsPage.tsx
components/features/clients/
  ├── ClientCard.tsx
  ├── ClientModal.tsx
  └── ClientDealsList.tsx
```

### Требования

**Deals / воронка:**

- DnD карточек → PATCH /api/deals/{id}/stage.  
- Фильтры: менеджер, источник, дата, сумма; суммы по стадиям.  
- Быстрое создание сделки; стадии и цвета из конфига воронки.  
- Отказы: `lost` + обязательный `lost_reason`; отдельный список/фильтр.

**Messages:**

- Список диалогов, непрочитанные, переписка по сделке.  
- Отправка в Telegram / Instagram; polling ~5 с; cursor-пагинация истории.  
- Медиа через API-прокси.

**Clients:**

- CRUD, поиск (имя, телефон, email, компания).  
- В карточке: сделки + переписки.  
- Привязка клиента к сделке; теги; экспорт CSV.  
- Телефон — нормализация на бэкенде ([ENTITIES.md](./ENTITIES.md)).

### Приёмочные критерии

- Стадии и won/lost — правила на сервере; откат без admin запрещён.  
- Отправка сообщения: понятная ошибка, если интеграция не настроена.  
- Дедуп входящих; `crm.chats` на отправку.  
- GET /api/clients?search=... на сервере.  
- Клиент с активными сделками — только архивация.

## 5. Финансы (`finance`)

### Что делает

Планирование бюджета (БДР), заявки на оплату, дебиторка, банковская выписка.

### Компоненты

```
components/FinanceView.tsx
  ├── FinanceBDR.tsx
  ├── FinanceRequests.tsx
  ├── FinanceReceivable.tsx
  └── FinanceStatements.tsx
components/modules/FinanceModule.tsx
```

### Требования

**БДР:** месяцы/кварталы/год; план vs факт; итоги на сервере.  

**Заявки:** state machine; `finance.approve`; фильтры; экспорт CSV.  

**Дебиторка:** overdue по `due_date`; частичное погашение → `partial`.  

**Выписка:** загрузка CSV/Excel; парсинг; ручное сопоставление с заявками.  

### Приёмочные критерии

- Суммы NUMERIC(15,2).  
- Approve только с правом на сервере.  
- Пагинация заявок ~50.

## 9. Настройки и система

### Что делает

Пользователи, права, **воронки**, **интеграции** (Meta, Telegram, MTProto, сайт), справочники, системные логи, DLQ. Интеграции как продукт описаны в [INTEGRATIONS.md](./INTEGRATIONS.md); здесь — **точка входа в UI**.

### Компоненты

```
components/SettingsView.tsx (lazy)
components/settings/
  ├── UsersSettings.tsx
  ├── FunnelsSettings.tsx
  ├── IntegrationsSettings.tsx
  ├── AutomationSettings.tsx
  ├── StatusesSettings.tsx
  └── SystemLogsSettings.tsx
```

### Требования

- Пользователи: CRUD, роли, granular permissions.  
- Воронки: стадии, источники, API-ключ сайта.  
- Интеграции: статус, MTProto, тест отправки.  
- Логи: system_logs, audit по сущности, DLQ — только admin.
