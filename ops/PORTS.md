# Порты, nginx, compose и статика

Краткая «истина» для деплоя и правок. Если твоё изменение **ломает** таблицу ниже или занимает **зарезервированный** порт — остановись и согласуй с владельцем инфраструктуры.

---

## Канонический ряд API на хосте (taska-семейство)

Один домен = один порт на `127.0.0.1` = один `BACKEND_PUBLISH_PORT` в `.env` этого клона. **Nginx `proxy_pass` обязан указывать на тот же порт.**

| Домен (пример) | `proxy_pass` / health | Compose-проект (`name:`) | Примечание |
|----------------|------------------------|---------------------------|------------|
| `tipa.taska.uz` | `127.0.0.1:8051` | `tipa` | контейнер `…-backend-1` |
| `chiranaasia.taska.uz` | `127.0.0.1:8052` | `chiranaasia` | |
| `unique.taska.uz`, `api-unique.taska.uz` | `127.0.0.1:8053` | `uniquetaskauz` (или как задано в том репо) | один бек на два имени |

**Новый стек того же класса:** следующий свободный порт **8054, 8055, …** (+1 к последнему занятому в этом ряду), без «прыжков», чтобы не плодить хаос.

Другие продукты на том же сервере (не смешивать с рядом 805x без необходимости):

| Домен | Порт |
|-------|------|
| `ecomonad.tipa.uz` | `127.0.0.1:9071` |
| `prince-club.tipa.uz`, `admin-prince-club.tipa.uz` | `127.0.0.1:9072` (upstream в nginx, например `prince_backend` → тот же порт) |

## Зарезервированы / не использовать под новые сервисы без явного ОК

Порты считаются занятыми снятыми или чужими стеками — **не поднимай** туда новый сервис без подтверждения:

`3020`, `3021`, `8004` (старый amicus), `3027` (russian-mamont), `3010` (uchetgram.uz), `5011` (urban210).

## Backend в Docker (taska / chiranaasia в этом репо)

| Где | Что |
|-----|-----|
| `docker-compose.yml` | `"${BACKEND_PUBLISH_PORT:-8003}:8000"` — **не подменяй** на жёсткий литерал; реальный порт только из `.env`. |
| `.env` на сервере | `BACKEND_PUBLISH_PORT=8051` / `8052` / … в соответствии с таблицей выше и nginx. |
| `ops/scripts/deploy.sh` | Читает `BACKEND_PUBLISH_PORT` для health-check — должен совпадать с compose и nginx. |

**502** на `/api` почти всегда: nginx смотрит не в тот порт или контейнер не слушает.

Проверка:

```bash
grep BACKEND_PUBLISH_PORT /var/www/<клон>/.env
curl -s -o /dev/null -w "%{http_code}\n" "http://127.0.0.1:<порт>/health"
docker compose -f /var/www/<клон>/docker-compose.yml ps
```

## Имя проекта Compose (`name:`)

В корне `docker-compose.yml` **обязательно** задано `name: tipa` (в этом репозитории) или в клоне chiranaasia — `name: chiranaasia`.

Без `name:` Compose возьмёт имя из каталога (`chiranaasia.taska.uz` → `chiranaasiataskauz`) и создаст **новые** volumes — связь со старыми данными потеряется.

**Не удаляй и не переименовывай `name:` без бэкапа volumes.**

---

## Nginx: два места, одна правда

1. **Боевой файл:** `/etc/nginx/sites-available/<домен>` (или как принято на хосте).
2. **Источник в репозитории:** `ops/nginx/nginx.conf` — деплой копирует/сверяет с sites-available.

После правок: `nginx -t`, затем **`nginx -s reload`** (не полагайся только на `systemctl reload nginx` — воркеры иногда не обновляются).

`proxy_pass` для каждого `server_name` должен совпадать с **`BACKEND_PUBLISH_PORT`** этого клона.

## Статика (фронт)

- Сборка: `<клон репо>/apps/web/dist/`.
- **`root` в nginx** для домена должен в итоге указывать на **dist этого клона**, а не на общий каталог, куда пишет «кто угодно» (иначе отдастся чужой билд).
- Если используется симлинк деплоя — он должен вести на **`…/apps/web/dist` этого проекта**, а не на разделяемый глобальный путь вроде «один `/var/www/frontend` на всех».

## Alembic (кратко)

- В одной ветке — **один** head; новая ревизия — `down_revision` на текущий head.
- Не создавай параллельные миграции, которые создают **одну и ту же таблицу** (пример конфликта: две ревизии создали `org_system_prefs` — прод ломается).
- При риске конфликта по таблице — миграция **идемпотентна**: перед `op.create_table` проверка `get_table_names()`.

---

## БД

- Не хардкодь `taska:taska` в `DATABASE_URL` для разовых команд — используй `docker compose run …`, чтобы подтянулся `.env`.
- Уважай `DB_PASSWORD` из `.env` клиента.

## prince-club / infra (другой репозиторий)

В compose маппинг `"<хост-порт>:4000"`; в nginx — `upstream … { server 127.0.0.1:<тот же порт>; }`. Согласованность как у taska: один порт на цепочку nginx → приложение.
